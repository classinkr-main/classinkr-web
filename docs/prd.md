# Classin Home PRD

기준 시점: 2026-03-19  
문서 목적: 현재 저장소 기준으로 `Classin Home` 랜딩/도입 사이트를 제품 관점에서 다시 정의하고, 구현 우선순위와 요구사항을 명확히 정리한다.

## 1. 제품 개요

### 제품명

- Classin Home

### 제품 성격

- 학원 운영 솔루션 `Classin`의 B2B 랜딩 및 리드 수집 웹사이트

### 현재 상태 요약

- 메인 랜딩, 제품 소개, 요금 계산, 블로그, 행사, 문의 페이지가 구현되어 있다
- 시각적 완성도는 높은 편이지만, 전환 흐름과 운영 안정성은 아직 보강이 필요하다
- 핵심 과제는 “예쁜 랜딩”을 “신뢰 가능한 리드 생성 제품”으로 바꾸는 것이다

## 2. 문제 정의

현재 사이트는 브랜드 인상과 메시지 전달은 잘 수행하지만, 다음 문제가 있다.

1. 일부 CTA가 실제 행동으로 이어지지 않는다
2. 블로그/이벤트 콘텐츠 모델은 있지만 운영 시스템은 없다
3. 리드 수집 API가 설정 누락 시에도 성공처럼 보일 수 있다
4. 분석 이벤트와 전환 데이터가 운영자 관점에서 신뢰하기 어렵다
5. 기획/디자인/개발이 공통으로 참조할 PRD 수준 문서가 부족하다

## 3. 목표

### 비즈니스 목표

1. 도입 문의 수를 안정적으로 늘린다
2. 상담 전환율이 높은 사용자를 더 빠르게 걸러낸다
3. 마케팅 비용 대비 전환 효율을 측정 가능하게 만든다

### 사용자 목표

1. 학원 원장/운영 책임자가 3분 안에 제품 가치를 이해한다
2. 관심 사용자가 망설이지 않고 문의 또는 자료 요청을 남긴다
3. 블로그/이벤트 방문자가 자연스럽게 전환 퍼널로 이동한다

### 제품 목표

1. 모든 주요 CTA는 실제 액션으로 연결된다
2. 리드 수집 성공/실패 상태가 운영자 기준으로 명확하다
3. 콘텐츠와 운영 데이터가 최소한 문서 또는 구조상 분리된다

## 4. 대상 사용자

### 1차 사용자

- 학원 원장
- 다지점 운영 책임자
- 교육 사업 본부장
- 학부모 커뮤니케이션/상담 운영 담당자

### 2차 사용자

- 마케팅 담당자
- 영업/CS 담당자
- 내부 운영팀

### 대표 페르소나

#### Persona A. 학원 원장

- 관심사: 재등록률, 강사 관리, 학부모 만족도
- 기대 행동: 제품 소개 확인 후 문의 제출

#### Persona B. 운영 책임자

- 관심사: 지점 표준화, 데이터 통합, 리포트 자동화
- 기대 행동: 제품/요금 페이지를 비교 후 데모 요청

#### Persona C. 마케팅 담당자

- 관심사: 캠페인 유입 성과, CTA 효율, 전환 이벤트 정확도
- 기대 행동: 이벤트/블로그 방문자를 문의로 연결

## 5. 성공 지표

### 핵심 지표

- 문의 제출 수
- 문의 제출 전환율
- CTA 클릭 대비 실제 제출율
- 문의별 소스 분포(`demo_modal`, `contact_page`, 추후 `newsletter`)

### 보조 지표

- 자료 다운로드 클릭 수
- 영상 보기 클릭 수
- 블로그/이벤트 페이지에서 문의 페이지로 이동한 비율
- 폼 제출 실패율
- 외부 웹훅 전송 성공률

### 운영 지표

- 환경변수 누락 상태에서의 오탐 성공 0건
- 실제 소스 기준 lint 오류 0건
- 문의 누락 사고 0건

## 6. 범위

## 6-1. MVP 범위

1. 메인 랜딩과 핵심 제품 메시지 유지
2. 문의 폼, 데모 모달, CTA 액션 정리
3. 리드 수집 API 안정화
4. 블로그/이벤트의 전환 연결성 보강
5. 최소 수준의 분석 이벤트 정확도 확보

## 6-2. 이번 PRD 범위 밖

- 실제 LMS/학원 운영 제품 본체 개발
- 관리자용 CMS 구축
- 회원가입/로그인/결제 시스템
- 완전한 블로그 CMS 마이그레이션
- 실시간 채팅 백엔드 구축

## 7. 사용자 흐름

### 핵심 퍼널

1. 사용자가 광고/검색/직접 유입으로 사이트 방문
2. 메인 랜딩 또는 제품 페이지에서 가치 제안 확인
3. CTA 클릭
4. 데모 요청 또는 문의 제출
5. `/api/lead` 를 통해 외부 운영 채널로 전달
6. 영업/운영팀이 후속 응대

### 보조 퍼널

1. 사용자가 블로그/이벤트 페이지 방문
2. 콘텐츠로 신뢰 확보
3. 뉴스레터/자료/문의 CTA로 이동
4. 문의 제출 또는 자료 요청

## 8. 기능 요구사항

## 8-1. 공통 레이아웃

### 요구사항

- 전역 헤더는 모든 주요 라우트에서 동일한 네비게이션 경험을 제공해야 한다
- 현재 페이지 위치가 명확하게 보이도록 active 상태가 표시되어야 한다
- 모바일에서도 주요 CTA가 접근 가능해야 한다

### 완료 기준

- 모든 페이지에서 헤더 동작 일관성 확보
- 푸터 정책 결정
  - 사용한다면 전역 연결
  - 사용하지 않는다면 제거

## 8-2. 메인 랜딩

### 요구사항

- Hero 섹션에서 제품 가치가 1스크린 내에 이해되어야 한다
- 메인 랜딩의 모든 CTA는 실제 액션으로 이어져야 한다
- 섹션 구성은 “문제 제기 → 해결 방식 → 근거 → 사례 → FAQ → 최종 CTA” 흐름을 유지한다

### 완료 기준

- Hero CTA 3종의 목적이 각각 명확함
  - 도입 문의
  - 자료 다운로드
  - 데모/투어 영상
- 최종 CTA는 단순 버튼이 아니라 실제 전환 수단을 제공

## 8-3. 제품 페이지

### 요구사항

- `/product/sw` 와 `/product/hw` 는 서로 다른 메시지를 제공하되, 동일한 문의 흐름으로 이어져야 한다
- 하드웨어/소프트웨어 각각의 CTA 목적이 명확해야 한다

### 완료 기준

- 탭 전환이 직관적임
- 버튼 클릭 시 실제 전환 경로 존재

## 8-4. 요금 페이지

### 요구사항

- 요금 계산기는 입력값에 따라 비교 가능한 결과를 제공해야 한다
- 추천 플랜 표시 기준이 설명 가능해야 한다

### 완료 기준

- 계산 결과 문구와 추천 기준이 사용자에게 이해 가능
- 추후 영업 응대 시 같은 기준으로 설명 가능

## 8-5. 블로그

### 요구사항

- 블로그 목록은 카테고리/검색이 가능해야 한다
- 상세로 이동 가능한 카드만 링크를 가져야 한다
- 뉴스레터 CTA는 실제 수집 플로우와 연결되어야 한다

### 완료 기준

- `/blog/[id]` 가 구현되거나 링크 제거
- 뉴스레터는 `alert` 수준을 벗어나 저장/전달 가능

## 8-6. 행사 / 프로모션

### 요구사항

- 이벤트 목록은 실제 데이터 기준으로 노출되어야 한다
- 더미 이미지와 임시 카피가 운영 환경에 남지 않아야 한다

### 완료 기준

- `events` 데이터 소스 명확화
- 이미지/일정/장소/CTA 링크 검증 완료

## 8-7. 문의 / 리드 수집

### 요구사항

- 문의 폼과 데모 모달은 공통 스키마를 사용해야 한다
- 최소한의 서버 검증이 있어야 한다
- 저장 실패를 성공처럼 보여주면 안 된다

### 완료 기준

- `LeadPayload` 입력 검증 추가
- 1개 이상 유효한 연동이 없으면 명시적 실패 처리
- 부분 실패와 전체 실패 구분
- 운영자가 유입 소스를 구분 가능

## 8-8. 분석 / 트래킹

### 요구사항

- CTA, 제출, 자료 다운로드, 영상 보기 이벤트는 채널별로 추적 가능해야 한다
- 퍼블릭 ID가 비어 있으면 잘못된 placeholder 값으로 추적하지 않아야 한다

### 완료 기준

- 분석 스크립트는 설정된 provider만 활성화
- `trackEvent` 타입 정리
- 전환 이벤트 명세 문서화

## 9. 비기능 요구사항

### 성능

- 주요 페이지는 정적 생성 유지
- 초기 진입 성능 저하를 막기 위해 무거운 섹션은 필요 시 동적 로드 유지

### 안정성

- 리드 누락이 없어야 한다
- 린트/빌드 기준이 신뢰 가능해야 한다

### 접근성

- 폼 필드 라벨 명확화
- 버튼/링크 역할 구분
- 모달 포커스 흐름 보장

### 유지보수성

- 대형 페이지 파일은 섹션 단위로 분해 가능해야 한다
- 콘텐츠 데이터는 페이지 로직에서 점진적으로 분리한다

## 11. 수용 기준

### 리드 수집

- 문의 제출 후 최소 1개 이상 외부 시스템에 전달되어야 한다
- 전달 실패 시 사용자/운영자 기준으로 실패가 식별 가능해야 한다

### CTA

- 주요 CTA 버튼에는 모두 목적지 또는 액션이 있어야 한다
- “트래킹만 되고 실제 동작은 없는 버튼”이 없어야 한다

### 콘텐츠

- 더미 데이터/임시 이미지/`href="#"` placeholder 가 운영 라우트에서 제거되어야 한다

### 품질

- `npx eslint app components lib` 통과
- `npm run build` 통과

## 12. 리스크

1. 운영 환경에서 웹훅이 비어 있어도 문제를 늦게 발견할 수 있다
2. 콘텐츠 확정이 늦으면 이벤트/블로그가 계속 임시 상태로 남을 수 있다
3. 분석 ID 관리가 느슨하면 마케팅 데이터 신뢰도가 떨어질 수 있다
4. 대형 파일 구조를 그대로 유지하면 수정 속도보다 회귀 위험이 커질 수 있다

## 13. 오픈 질문

1. 자료 다운로드의 실제 산출물은 무엇인가
2. “3분 투어 영상”은 외부 영상인지, 자체 모달 재생인지
3. 블로그는 상세 페이지를 만들 것인지, 목록형 콘텐츠 허브로 둘 것인지
4. 뉴스레터는 어떤 도구로 수집/전송할 것인지
5. 푸터를 실제 사이트 정책 정보로 채울 것인지
6. 운영 연락처 정보는 코드가 아닌 설정 파일로 뺄 것인지

## 14. 제안 일정

### Phase 1. 전환 안전화

- 리드 API 수정
- CTA 실동작 연결
- 블로그 404 제거

### Phase 2. 콘텐츠 운영화

- 이벤트 더미 데이터 제거
- 뉴스레터 실제 연결
- 푸터/정책 링크 정리

### Phase 3. 구조 개선

- analytics 타입 정리
- 대형 페이지 파일 분리
- 콘텐츠 데이터 계층 분리

## 15. 한 줄 정의

`Classin Home`은 단순 소개 사이트가 아니라, 학원 운영 솔루션의 신뢰를 만들고 실제 상담 기회를 수집하는 “리드 전환 제품”이어야 한다.
